Node-first rewrite: what changed in the method sections

Every edit below replaces the old speed-first wording with the adopted node-first formulation. The old version is kept in the source, wrapped in \begin{comment} (hidden from the PDF); the new version sits right beside it, active. Nothing was deleted.

The one idea behind all of it

The DP no longer chooses speed from a finite set V (the old 61-value grid). Speed is a continuous variable on the interval 𝒱 = [v_min, v_max]; the DP instead discretises the time–distance plane onto the ζ, τ grid and reads speed off as v̄ = Δd/Δt. So the ζ/τ grid is the only discretisation, and the finite speed set disappears — which also fixes a latent §3↔§4 mismatch (§3 already defined 𝒱 as continuous).

§4.2.1 · Discretising the free coordinate

2 edits

Why the naive state set is intractable

commented
… since the speed of the vessel is selected from a finite set $V$, the other
coordinate can also obtain a value from a finite set. Therefore the set of
reachable states is finite --- $V$ is finite and the number of lines is finite
--- but its cardinality can grow as $|V|^{(lines crossed)}$, which is not tractable.
active
Any state at which the DP makes a decision lies on a distance line or a time
line … The other, free coordinate is continuous: the speed is drawn from the
interval $\mathcal{V}$, so from a single state the vessel can reach a continuum
of points on the next line, and over the whole voyage the reachable set is
infinite --- and even a naive discretisation of the speed into $K$ values would
let the state count grow as $K^{(lines crossed)}$, which is not tractable.

Reframes the blow-up: it is no longer “because |V| is finite” but “because speed is continuous (a continuum of successors), and even a crude K-value speed grid would explode as K^(lines)” — motivating the grid discretisation without ever committing to a speed set.

What snapping does · arc-count bound, Eq. (6)

commented
Snapping merges the many near-coincident arrivals produced by distinct speed
sequences into a single shared state … giving

active
The grid nodes on each line are then the discrete states, and from a state we
enumerate directly the reachable grid nodes on the next line ahead --- so no
speed set is ever introduced. … giving
   |S| = O(M·T/τ + Θ·L/ζ),   |A| = O(K·|S|),
where A is the induced arc set and K is the number of far-wall grid nodes

The grid nodes are the states/actions. Old: snapping merged arrivals from many speeds, and each state emitted up to |V| arcs → bound O(|V|·|S|). New: enumerate the reachable grid nodes directly, at most K per state → O(K·|S|). On Route 1, K ≈ 8 vs the old |V| = 61.

§4.2.2 · Enumerating the reachable states

2 edits
8

How successors are generated (prose)

commented
Choosing a speed $v\in V$, the vessel holds $v$ until it first meets one of these
two lines … terminates at the nearer. Its free coordinate is snapped … giving a
successor (d',t'); the realised speed over the arc is v̄ = Δd/Δt …
active
Rather than iterate over a set of speeds, we enumerate the reachable grid nodes
on these two lines directly. On the next distance line, holding any speed in 𝒱
gives an arrival time between t+(d̄−d)/v_max and t+(d̄−d)/v_min (clipped at t̄ and
T); we take every multiple of τ. On the next time line, the reachable distance
lies between d+v_min·δt and d+v_max·δt; we take every multiple of ζ. Each grid
node is a successor reached at the single speed v̄ = Δd/Δt ∈ 𝒱 … because the node
already lies on the grid, no target-versus-realised speed gap arises. A distance
line too close to resolve on the τ-grid is skipped, and the vessel glides past
it to the next time line.

The core mechanic. Old: pick a speed, follow it to a line. New: enumerate the distinct reachable grid nodes on the far wall (arrival times on a distance line, arrival distances on a time line) and take the speed as Δd/Δt. This removes the target-vs-realised speed gap and adds the corner rule for a too-close cell boundary.

Algorithm 1 — the build loop

commented · speed-first
for (d,t) popped from Q:
  d̄ ← next distance line;  t̄ ← next time line
  for v ∈ V:                          ◄── loop over the 61-value speed set
    δd ← (d̄−d)/v;  δt ← t̄−t
    if δd ≤ δt: (d',t') ← (d̄, ⟨t+δd⟩_τ)        ▷ distance line binds
    else:       (d',t') ← (⟨d+v·δt⟩_ζ, t̄)      ▷ time line binds
    v̄ ← Δd/Δt;  add arc (d,t)→(d',t') with cost φ·Δt
active · node-first
for (d,t) popped from Q:
  d̄ ← next distance line;  t̄ ← next time line;  C ← ∅
  res ← ⟨t+(d̄−d)/v_max⟩_τ > t          ◄── is the distance line τ-resolvable?
  if res:                                        ▷ enumerate arrival TIMES on d̄
    for k = round((t+(d̄−d)/v_max)/τ) .. round(min(t+(d̄−d)/v_min, t̄, T)/τ):
      add (d̄, kτ) to C
  d_cap ← d̄ if res else L                        ▷ enumerate arrival DISTANCES on t̄
  for k = round((d+v_min·δt)/ζ) .. round(min(d+v_max·δt, d_cap)/ζ):
      add (kζ, t̄) to C
  for (d',t') ∈ C:
    v̄ ← Δd/Δt;  skip if v̄ ∉ 𝒱 or unattainable;  add arc with cost φ·Δt

The engine that produced §6’s numbers. The for v ∈ V loop is replaced by direct enumeration of candidate far-wall grid nodes, with the corner rule (d_cap = L when the distance line is too close, so the vessel glides past it). Input is now a speed band [v_min, v_max], not a speed set.
